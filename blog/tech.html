Test Driven Development


Many definitions for Test Driven Development exist across developer blogs and read something like this... "Test Driven Development is the process of developing software applications by writing tests before and during the course of developement, to ensure that source code is funcctional before it is complete." After reading up on the benefits and the reasoning behind TDD, I think of it a little more simply. TDD, in the opinion of this burgeoning developer, is a system of developing code according to expectation. That's probably over-simplifying, but here's my thinking...

So far at Dev Bootcamp, we've seen several examples of taking a program from the early steps of a development idea (a user story, for example) to pseudocode - an English, non-coding-language-specific description of how to write the code that will accomplish the task described in the user stories. From there (and we're assuming we don't already have test-code provided for us), it's a matter of writing code that will solve the problem. But this may be too hasty. What if in between the pseudocode and the initial solution, we had another way to break down the process for us into even smaller steps. What if we had a way to test our expectations of the code? Test Driven Development lets us do exactly that.

As a new developer, it's been easy to try every new method, syntactic pattern or trick I've learned in the hunt for a solution to a given problem. Before too long, I've got a whole solution that only half works. TDD is exactly what someone like me needs to reign it in. In TDD, you start with a test that states what you expect your program to do, little by little. Examples of this that we've seen so far would be "assert" in Javascript, "describe" and "expect" in Ruby, and others. What these built in methods of TDD allow us to do is state the expectation of a particular expression of code based on a supposed input, and allow us to shape our code based on the result of the test.

The process goes something like this...

  • Write a test that states what one portion of code should do
  • Run that test. It should fail, because you haven't written working code yet
  • Write the code that will make that test pass
  • Run the test
  • If it failed, fix it. Make the test pass before you write anymore code or tests
  • If it passed, clean it up - refactor it
  • After refactoring, if it still works, write your next test, and repeat the process
This is a very basic description, but what I think is most important about this entire philosophy of development is the fact that at every step along the way, you are making sure that you are implementing working code. Your time spent squishing bugs, troubleshooting, and banging your head against the wall after you've written a few hundred (or thousand) lines of code will be greatly minimized. Also, you're starting small, and taking small steps along the way. Most great challenges are accomplished not by shooting straight for the largest goal, but by accomplishing many smaller goals on the way to the larger one. It would seem to me that development is no different.
